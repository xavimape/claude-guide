Plataforma 01

Claude en la Web

Accedé a claude.ai desde cualquier navegador. Es el punto de entrada más completo para análisis de documentos, investigación con web search y flujos que no requieren acceso al código fuente local.

💬

Proyectos y Contexto Persistente

Organizá tu trabajo en Proyectos con instrucciones de sistema propias, archivos adjuntos y historial persistente entre sesiones. Ideal para flujos repetitivos de equipo.

🔍

Web Search & Deep Research

Activá la búsqueda web para respuestas actualizadas. "Deep Research" sintetiza decenas de fuentes y genera informes estructurados. Ideal para threat intel y papers técnicos.

📄

Análisis de Archivos

Subí PDFs, imágenes, CSVs, logs, código fuente. Claude puede comparar múltiples documentos, extraer datos estructurados y generar visualizaciones con Code Execution.

⚙️

System Prompt por Proyecto

Configurá un system prompt que define el rol, tono y restricciones de Claude para toda la sesión. Ejemplo: "Actuás como analista SOC Nivel 1 especializado en SIEM OSSIM..."

🎨

Artifacts Interactivos

Claude puede generar componentes React, diagramas SVG, dashboards HTML y widgets interactivos que se renderizan en tiempo real. Útil para visualizar datos de seguridad.

🧠

Memoria entre Sesiones

Habilitá la memoria en Configuración para que Claude recuerde tus proyectos activos, herramientas favoritas y contexto de trabajo entre conversaciones distintas.

Técnicas de prompting eficiente
Rol + Contexto
Siempre definí el rol al inicio. "Actuás como analista de ciberseguridad con certificación OSCP, especializado en análisis de logs SIEM. Revisá esto..." produce resultados notablemente mejores que preguntas genéricas.
Formato explícito
Pedí el formato de salida. "Respondé en JSON con claves: vulnerability, cwe_id, severity, affected_line, fix" elimina ambigüedades y hace la respuesta inmediatamente usable en pipelines.
Chain-of-thought
Agregá "Pensá paso a paso antes de responder" para análisis complejos. Claude externaliza el razonamiento y produce análisis más precisos y verificables.
Ejemplos few-shot
Incluí 2-3 ejemplos concretos del input/output esperado. Esto calibra el estilo y nivel de detalle mejor que cualquier descripción en prosa.
Restricciones negativas
Decile qué NO hacer. "No incluyas explicaciones genéricas, solo hallazgos concretos con número de línea" acota el espacio de respuesta de forma efectiva.
Tags XML
Estructurá prompts largos con XML. <context>...</context> <task>...</task> <format>...</format> mejora la separación entre instrucciones y reduce alucinaciones.
💡
Truco pro: Instrucciones en XML

Para prompts complejos de seguridad usá <scope>...</scope> <code>...</code> <criteria>OWASP Top 10</criteria> <output>JSON</output>. Claude procesa cada sección de forma aislada y produce salidas más estructuradas.

Plataforma 02

Claude Desktop para Windows

La app de escritorio une las tres experiencias de Claude en un solo cliente nativo: conversación general, automatización de archivos (Cowork) y la potencia completa de Claude Code con CI/CD integrado.

Sub-sección A

Chat

El chat en Desktop tiene las mismas capacidades que la web, pero con acceso nativo al portapapeles del sistema, hotkeys globales y notificaciones de Windows al terminar tareas largas.

⌨️

Atajo global

Configurá un hotkey (ej. Alt+Space) para invocar Claude desde cualquier app Windows sin perder el foco. Ideal para consultas rápidas durante análisis de seguridad.

📋

Paste inteligente

Copiá cualquier log, stack trace o output de terminal desde otra app y pegalo en el chat. Claude reconoce el contexto automáticamente y ofrece acciones relevantes sin que lo especifiques.

🗂️

Proyectos sincronizados

Todos tus Proyectos de claude.ai están disponibles en Desktop. Continuá conversaciones desde el móvil, web y Desktop de forma transparente.

🔔

Notificaciones nativas

Recibís notificaciones de Windows cuando Claude termina tareas largas. Lanzá un análisis de logs, minimizá la ventana y seguí trabajando: Claude te avisa al terminar.

Sub-sección B

Cowork — Automatización de escritorio

🤖
¿Qué es Cowork?

Automatizá tareas de archivos, procesamiento de documentos y pipelines sin escribir código. Para analistas de seguridad: procesá logs en lote, organizá reportes de auditoría y generá resúmenes automáticos de CVEs.

Cowork — Casos de uso en seguridad
# Consolidar logs de múltiples fuentes en un reporte Cowork: "Leé todos los .log de /var/log/ossim/, filtrá eventos con severity=HIGH o CRITICAL, y generá /Reportes/alerta-diaria.csv con columnas: timestamp, src_ip, event_type, count" # Organizar reportes de pentesting por cliente Cowork: "Procesá los PDFs de /Pentests/2024/, extraé nombre del cliente, fecha y severidad general de cada reporte, y creá un índice en /Reportes/indice-pentests.xlsx" # Renombrar evidencias con formato estándar Cowork: "Renombrá todas las capturas de /Evidencias/ con formato AAAA-MM-DD_cliente_hallazgo-N usando el contenido del archivo."
Sub-sección C

Claude Code — CLI + CI/CD + Docker

Claude Code es el agente de codificación de Anthropic. Entiende tu repositorio completo, ejecuta comandos y trabaja de forma autónoma. Integrable en GitHub Actions y Docker.

PowerShell / Terminal
# Requisito: Node.js 18+ node --version # debe ser v18 o superior # Instalación global via npm npm install -g @anthropic-ai/claude-code # Verificar instalación claude --version # Primer uso — autenticación claude > Iniciá sesión en claude.ai cuando el browser se abra... # Inicio en un proyecto existente cd C:\mis-proyectos\mi-app claude # Modo no-interactivo (para scripts / CI) claude -p "Describí la arquitectura de este proyecto" --output-format json
claude "tarea"
Modo agente completo. Claude analiza el repo, planea, escribe código, ejecuta tests y valida. Ej: claude "Revisá src/ buscando SQL injection y generá fixes"
claude -p "..."
Print mode (no-interactivo). Ejecuta una sola consulta y retorna. Soporta --output-format json para pipelines CI.
claude --continue
Continuá la última sesión. Retoma el contexto anterior sin re-explicar el proyecto.
claude --resume [id]
Retomá una sesión específica por su ID. Listá sesiones con claude sessions list
CLAUDE.md
Archivo de contexto del proyecto. Claude lo lee automáticamente al iniciarse. Documentá la arquitectura, convenciones, comandos críticos y criterios de seguridad.

Los hooks ejecutan scripts automáticamente antes o después de acciones de Claude. Se configuran en .claude/settings.json.

.claude/settings.json — Hooks de seguridad
{ "hooks": { // Lint + security scan ANTES de que Claude escriba archivos "PreToolUse": [{ "matcher": "Write", "hooks": [{ "type": "command", "command": "npx eslint --fix $FILE && npx semgrep --config auto $FILE" }] }], // SAST scan después de cada write "PostToolUse": [{ "matcher": "Write", "hooks": [{ "type": "command", "command": "npx snyk test --file=$FILE 2>> security.log || true" }] }], // Notificación y resumen al finalizar sesión "Stop": [{ "hooks": [{ "type": "command", "command": "powershell -c \"[System.Media.SystemSounds]::Beep.Play()\"" }] }] } }
🔒
Hooks como guardia de seguridad

Los hooks en PreToolUse pueden bloquear acciones antes de que se ejecuten. Usá esto para prevenir que Claude elimine archivos críticos o guarde código sin pasar un SAST scan primero.

Los servidores MCP (Model Context Protocol) extienden Claude Code con herramientas externas: bases de datos, APIs, servicios de seguridad, y más.

Configurar MCP — claude mcp add
# MCP de GitHub claude mcp add github --transport sse "https://mcp.github.com/sse" # MCP de PostgreSQL (local) claude mcp add postgres --transport stdio \ "npx @modelcontextprotocol/server-postgres" \ --env DATABASE_URL="postgresql://user:pass@localhost/db" # MCP de Shodan (inteligencia de amenazas) claude mcp add shodan --transport stdio \ "npx @shodan/mcp-server" \ --env SHODAN_API_KEY="tu-api-key" # Ahora Claude puede usar herramientas en conversación: > "Buscá en Shodan hosts con el puerto 8080 abierto en el rango 192.168.1.0/24" Usando herramienta: shodan.host_search ✓ Encontrados 3 hosts — exportando a hosts-exposed.json

Claude Code puede integrarse en pipelines de GitHub Actions para code review automatizado, análisis de seguridad en PRs y generación de documentación.

.github/workflows/claude-security-review.yml
name: Claude Security Review on: pull_request: types: [opened, synchronize] jobs: claude-review: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 with: fetch-depth: 0 - name: Install Claude Code run: npm install -g @anthropic-ai/claude-code - name: Get changed files id: changed run: | echo "files=$(git diff --name-only origin/main HEAD | tr '\n' ' ')" >> $GITHUB_OUTPUT - name: Claude OWASP Review env: ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }} run: | claude -p "Revisá los archivos modificados en este PR: ${{ steps.changed.outputs.files }} Aplicá criterios OWASP Top 10 2023 + principios SOLID. Para cada hallazgo: archivo, línea, CWE, severidad (CRITICAL/HIGH/MEDIUM/LOW), fix. Salida: JSON array." --output-format json > review.json - name: Post Review as PR Comment uses: actions/github-script@v7 with: script: | const fs = require('fs'); const review = JSON.parse(fs.readFileSync('review.json', 'utf8')); const body = review.map(r => `**${r.severity}** · ${r.file}:${r.line} · ${r.cwe}\n> ${r.fix}` ).join('\n\n'); github.rest.issues.createComment({ ...context.repo, issue_number: context.issue.number, body });

CLAUDE.md para CI/CD

En repos con GitHub Actions, agregá al CLAUDE.md una sección ## CI Rules con las reglas específicas del pipeline: qué archivos no tocar, qué tests correr siempre, y qué severidades bloquean el build.

El archivo CLAUDE.md — System prompt de repositorio

CLAUDE.md — Estructura recomendada
# Mi Proyecto ## Stack - Backend: Node.js 20 + Express + TypeScript - DB: PostgreSQL 16 con Prisma ORM - Tests: Vitest + Testing Library ## Comandos críticos - `npm run dev` → servidor puerto 3000 - `npm run test` → tests (SIEMPRE antes de commit) - `npm run lint` → ESLint + Prettier ## Criterios de seguridad - Todo input del usuario debe pasar validación Zod antes de usarse - Queries a la DB: solo Prisma ORM, nunca SQL crudo - Secretos: siempre en .env, nunca hardcodeados - Auth: JWT con expiración 1h, refresh token en httpOnly cookie ## Convenciones - async/await (nunca .then()) - Errores manejados con Result<T,E> (ver src/types/result.ts) - Toda función pública necesita JSDoc con @throws

Configuración B

Comandos con /

/init
Inicializar el proyecto. Claude analiza el repositorio y genera un CLAUDE.md con arquitectura, stack detectado y convenciones. Hacelo siempre en proyectos nuevos.
/review
Code review completo. Claude revisa los cambios actuales con la profundidad de un senior dev: seguridad (OWASP), rendimiento, mantenibilidad y edge cases.
/test
Generar tests. Analiza el código activo y genera tests unitarios, de integración y edge cases usando el framework detectado. Incluye casos de seguridad.
/explain
Explicar código seleccionado. Descripción detallada de qué hace, por qué y cuáles son las implicaciones de rendimiento y seguridad.
/fix
Fix automático. Seleccioná código roto o el error del terminal y /fix diagnostica, propone la corrección con explicación del root cause.
/compact
Comprimir el contexto. Cuando el contexto se llena, crea un resumen eficiente de la sesión sin perder información clave. Usalo cada ~30 intercambios.
/doctor
Diagnóstico del entorno. Verifica que Claude Code tenga todos los permisos, herramientas y configuraciones necesarias para funcionar correctamente.

@url
Contenido de una URL. Ej: @https://owasp.org/Top10/ Implementá las mitigaciones relevantes en @src/

En VS Code: @git "Generá el CHANGELOG.md para la versión v2.1.0 basado en los commits desde v2.0.0. Clasificá en: — ✨ Features: nuevas funcionalidades — 🐛 Bug Fixes: correcciones — ⚡ Performance: mejoras de rendimiento — 🔒 Security: parches de seguridad (IMPORTANTE: listá al principio) — 💥 Breaking Changes: cambios que rompen compatibilidad — 🔧 Internal: refactoring, deuda técnica Seguí el formato keepachangelog.com. No incluyas commits de merge ni 'fix typo'.

🚨 Post-mortem de incidente de seguridad

Redactá un post-mortem blameless del siguiente incidente: <incident> [Describí el incidente: qué pasó, cuándo, qué sistemas afectó, cuánto duró] [Incluí los logs y alertas relevantes] </incident> Estructura del post-mortem: — Resumen ejecutivo: 3 líneas (qué pasó, impacto, resolución) — Timeline: cronología exacta con timestamps (detección → respuesta → resolución) — Root cause: causa raíz técnica (NO culpar personas) — Contributing factors: factores que lo hicieron posible o lo agravaron — Impact: usuarios afectados, datos, SLA, costo estimado — Resolution: qué se hizo para mitigarlo — Action items: tabla con Owner / Fecha / Prioridad para prevenir recurrencia — Lessons learned: qué salió bien y qué mejorar en el proceso de respuesta
